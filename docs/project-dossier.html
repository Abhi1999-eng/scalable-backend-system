1. Executive Summary

The project has evolved from a backend systems learning project into a practical ingestion-and-analytics platform. It now combines classic distributed backend components with a data-focused user experience:

  • Spring Boot REST APIs for operational and analytics workflows
  • PostgreSQL for durable metadata and dataset row storage
  • Redis for caching, rate limiting, and job progress state
  • Kafka for user and job event pipelines
  • Asynchronous executors for ingestion workloads
  • A Next.js frontend that uploads datasets, explores CSVs, previews tables, and builds charts

There are two major ingestion domains in the codebase:

  1. Bulk file ingestion: optimized for large batches of files or ZIP archives containing many files.
  2. Dataset analytics: optimized for CSV uploads, inferred profiling, row browsing, and chart-ready aggregations.

2. Goals and Problem Statement

The project solves several related problems:

  • Accept large upload workloads without blocking HTTP request threads.
  • Persist uploaded files and structured dataset content safely.
  • Expose progress tracking for long-running work.
  • Support common operational backend patterns such as caching, rate limiting, and event-driven invalidation.
  • Allow a user to upload multiple CSVs or ZIPs of CSVs, inspect each CSV independently, and create charts from it.
  • Provide a frontend workflow that makes analytical exploration approachable instead of overly technical.

5. Top-Level System Architecture

System shape: the application is a multi-workload platform with a shared operational backend and two specialized ingestion paths: one for generic file throughput and one for structured CSV analytics.

5.1 Main architectural components

  • Presentation tier: Next.js dashboard that handles uploads, dataset selection, previewing, and chart interactions.
  • API tier: Spring Boot controllers exposing REST endpoints for health, users, jobs, bulk file ingestion, and datasets.
  • Service tier: domain logic for caching, rate limiting, Kafka publishing/consuming, file ingestion, and dataset analytics.
  • Persistence tier: PostgreSQL tables for durable domain data plus filesystem storage for raw uploaded files.
  • Operational state tier: Redis for request limiting, cache entries, and transient job progress counters.
  • Event tier: Kafka topics for user events and job-step messages.
  • Async execution tier: dedicated thread pools for ingestion orchestration and ingestion workers.

6.2 Async execution model

AsyncConfig.java defines two executors:

  • ingestionTaskExecutor: core pool 4, max pool 8, queue 100. Used for orchestration of high-level ingestion tasks.
  • ingestionWorkerExecutor: core pool 8, max pool 16, queue 500. Used for parallel work over file chunks.

This separation is important. It prevents a single giant upload from monopolizing the same pool used to schedule new ingestion work.

7. Backend Modules and Responsibilities

7.1 Health module

HealthController provides a simple liveness endpoint. This supports quick local verification and frontend "backend online" indicators.

7.2 User, cache, and rate-limiting module

Relevant classes:

  • UserController
  • CacheService
  • RateLimiterService
  • UserRepository
  • User
  • KafkaProducerService and KafkaConsumerService

User API behavior

  • POST /users?name=... creates a user if the caller is under the rate limit.
  • GET /users/{id} tries Redis first, then PostgreSQL, then caches the result for 10 minutes.
  • DELETE /users/{id} deletes a user and publishes a Kafka event.
  • GET /users lists all users.

Rate limiting logic

The rate limiter is simple and Redis-backed. For a given client-id header:

  • Redis key: rate_limit:<clientId>
  • Window: 200 seconds
  • Limit: 15 requests

If the request count exceeds the threshold, the controller returns HTTP 429.

Caching logic

  • Redis key: user:<id>
  • TTL: 10 minutes
  • Cache is invalidated on delete by Kafka consumer logic

Event-driven invalidation

When a user is created or deleted, a JSON event is published to Kafka topic user-events. The consumer listens to that topic and clears the Redis user cache on delete events.

7.3 Background jobs module

Relevant classes:

  • JobController
  • JobStatusController
  • KafkaProducerService
  • JobConsumer

The job subsystem demonstrates event-driven background progress tracking:

  1. POST /jobs/start creates a pseudo job ID and sets Redis keys for total and progress.
  2. It publishes 100 messages to Kafka topic job-topic.
  3. A job consumer processes messages and updates progress.
  4. GET /jobs/{jobId} reads Redis and returns progress totals.

This part of the system is less feature-rich than the dataset subsystem, but it demonstrates asynchronous processing and distributed state tracking clearly.

7.4 Bulk file ingestion module

Relevant classes:

  • FileIngestionController
  • FileIngestionService
  • IngestionStatusService
  • StoredFileRecord
  • FileRecordRepository and FileRecordJdbcRepository

Purpose

This subsystem is optimized for volume, not structured analytics. Its job is to receive a large number of files, store them efficiently, compute metadata, and expose processing status.

Core design choices

  • Requests return quickly with an uploadId rather than processing everything synchronously.
  • Multipart uploads are staged before async processing so servlet temp-file cleanup cannot break background work.
  • ZIP ingestion streams entries instead of exploding memory usage.
  • File metadata writes are batched.
  • Files are copied with a 64 KB buffer to avoid loading entire contents into memory.

Important implementation constants

Constant Value Effect
DB_BATCH_SIZE 500 Controls how many file metadata records are batch inserted at once
FILE_CHUNK_SIZE 250 Controls multipart work chunking across async worker tasks
BUFFER_SIZE 64 KB Controls stream buffer size during file copy and digest calculation

Metadata captured per stored file

  • Upload ID
  • Original filename
  • Stored filename
  • Storage path
  • Content type
  • Size in bytes
  • Line count
  • SHA-256 checksum
  • Status
  • Creation timestamp

Bulk file ingestion flow

  1. Client submits multipart files or a ZIP archive.
  2. Service creates an uploadId and an initial status object.
  3. Input files are staged under a hidden staging directory.
  4. Async task creates final upload directory under uploads/<uploadId>.
  5. Files or ZIP entries are streamed to disk.
  6. Checksums and line counts are computed during the copy.
  7. Metadata records are batch inserted into PostgreSQL.
  8. Progress and byte counters are updated as files complete.
  9. Staging content is deleted after processing.

7.5 Dataset analytics module

Relevant classes:

  • DatasetController
  • DatasetService
  • Dataset
  • DatasetColumnProfile
  • DatasetRow
  • DatasetChartRequest and DatasetChartResponse
  • DatasetRepository, DatasetColumnProfileRepository, DatasetRowRepository

Purpose

This subsystem turns uploaded CSVs into first-class analytical datasets. It stores raw files, profiles columns, stores rows, and exposes chart-ready transformations for frontend visualization.

Key product capabilities

  • Upload one or many CSVs.
  • Upload ZIP archives that contain multiple CSVs.
  • Group all CSVs discovered in one upload under a shared batchId.
  • Track each CSV independently with its own datasetId.
  • Infer column type and profile statistics.
  • Store rows for preview and analytical aggregation.
  • Generate bar, line, pie, histogram, scatter, box, pictograph, and area chart payloads.

Dataset ingestion flow

  1. Client sends CSV files and/or ZIP archives to POST /datasets/upload.
  2. Service creates a shared batchId.
  3. Each CSV becomes a separate dataset record with its own datasetId.
  4. Raw dataset file is stored under dataset-uploads/<datasetId>/<filename>.
  5. Dataset status is saved as QUEUED.
  6. Async executor starts ingestDataset(datasetId).
  7. CSV headers are read and rows are streamed through Apache Commons CSV.
  8. Each row is converted to JSON and persisted into dataset_rows.
  9. Per-column accumulators compute distinct counts, null counts, and min/max snapshots.
  10. Column profiles are saved into dataset_column_profiles.
  11. Dataset status becomes READY or FAILED.

Type inference strategy

Inference is rule-based, not AI-based. The service observes values column by column and classifies data into types like number, text, boolean, date, and datetime using deterministic parsing checks.

Chart generation strategy

Chart APIs do not return image files. They return chart-ready data structures. The frontend is responsible only for rendering. The backend decides how rows are grouped, aggregated, and reduced.

Chart Type Backend Strategy
Bar / Pie / Pictograph Category aggregation by xColumn, with count or sum
Line / Area Ordered aggregation across the requested X dimension
Histogram Distribution/bucketization over a numeric column
Scatter X/Y numeric points extracted from row data
Box Plot Min/Q1/median/Q3/max statistics over a numeric distribution

Why rows are stored as JSON

The current system persists each dataset row as JSON text in PostgreSQL. This is a pragmatic choice:

  • the schema of uploaded CSVs is not fixed ahead of time
  • one storage model can support arbitrary uploaded datasets
  • the backend can reconstruct maps for row preview and aggregation logic

This is a flexible MVP design. For much larger analytical workloads, a dedicated analytical engine such as DuckDB or ClickHouse would likely be a better next step.

10.4 Proxy architecture

Browser requests are sent to Next.js under /api/.... The catch-all route handler forwards those requests to the Spring backend using the configured BACKEND_URL. This avoids CORS complexity and lets the browser stay same-origin against port 3000.

10.5 Frontend interaction model

  1. User uploads CSVs and ZIPs in a single batch.
  2. Frontend receives a batchId and dataset list.
  3. Frontend polls batch state until queued datasets finish processing.
  4. CSV rows are shown as expandable cards.
  5. Opening a CSV loads the row preview and enables chart generation for that dataset.
  6. Chart data is rendered client-side using Recharts.

11. Detailed Data Flows

11.1 User creation and cache lifecycle

Client
  -> POST /users?name=...
  -> RateLimiterService checks Redis key rate_limit:<client-id>
  -> CacheService saves user in PostgreSQL
  -> KafkaProducerService publishes USER_CREATED event
  -> Response returns created user

Later read:
Client
  -> GET /users/{id}
  -> CacheService checks Redis user:<id>
  -> cache hit: return cached value
  -> cache miss: read PostgreSQL, write Redis with 10-minute TTL, return value

Delete:
Client
  -> DELETE /users/{id}
  -> CacheService deletes from PostgreSQL
  -> KafkaProducerService publishes USER_DELETED event
  -> KafkaConsumerService consumes event and invalidates Redis user:<id>

11.2 Bulk file ingestion flow

Client
  -> POST /ingestion/files/bulk or /ingestion/files/zip
  -> FileIngestionService creates uploadId and initial status
  -> multipart files or archive are staged under uploads/.staging/<uploadId>
  -> async task starts processing
  -> final files are streamed to uploads/<uploadId>
  -> checksum, line count, and size are computed
  -> metadata is batch inserted into PostgreSQL
  -> status is updated as processed bytes and files increase
  -> final status becomes COMPLETED or FAILED

11.3 Dataset analytics flow

User uploads CSVs or ZIPs
  -> POST /datasets/upload
  -> DatasetService creates batchId
  -> each CSV becomes a Dataset row with its own datasetId
  -> raw CSV is saved under dataset-uploads/<datasetId>
  -> async ingestion begins

Ingestion
  -> CSV headers parsed
  -> rows converted to ordered maps
  -> rows persisted as JSON in dataset_rows
  -> column accumulators infer types and profile metrics
  -> profiles written to dataset_column_profiles
  -> dataset row_count / column_count updated
  -> dataset status becomes READY

Exploration
  -> frontend loads batch
  -> user expands one CSV row
  -> frontend requests profile and rows
  -> user chooses chart type and columns
  -> frontend POSTs chart request
  -> backend returns labels/values or points/statistics
  -> Recharts renders visualization

13. Operational Characteristics

13.1 Performance characteristics

  • File ingestion is optimized for throughput by using async executors, file chunking, and JDBC batching.
  • Dataset ingestion is optimized for flexibility and deterministic profiling, not warehouse-scale OLAP performance.
  • ZIP uploads are the preferred high-volume path for file ingestion because they minimize HTTP part overhead.
  • Dataset row retrieval is capped and paged to keep preview endpoints manageable.

13.2 Security and safety controls present

  • ZIP path sanitization prevents path traversal during file ingestion.
  • Staging avoids temp-file lifetime bugs during async processing.
  • Rate limiting protects user APIs from simple burst abuse.
  • Chart request validation returns explicit 400 errors for malformed requests.

13.3 Notable technical debt and limitations

  • Schema evolution relies on Hibernate ddl-auto=update instead of Flyway or Liquibase.
  • Logging is still basic and includes printStackTrace() style behavior in Kafka services.
  • Dataset storage uses PostgreSQL row JSON rather than a purpose-built analytical engine.
  • Authentication and authorization are not implemented.
  • Object storage such as S3 or MinIO is not yet used; raw files are local-disk based.

14. Scalability Assessment

14.1 What scales reasonably well today

  • Bulk file receipt and metadata persistence for large batches
  • Async batch submission and progress tracking
  • Multi-dataset upload batches containing multiple CSVs or ZIP archives
  • Low-to-moderate scale CSV analytics and charting

14.2 Where scaling pressure will appear first

  • Large analytical queries over many stored JSON rows in PostgreSQL
  • Lack of schema migrations in multi-environment deployments
  • Local filesystem storage in distributed or cloud-native deployments
  • Frontend rendering if dataset previews become too wide or too large without virtualization

14.3 Best next evolution path

  1. Add Flyway migrations for repeatable schema management.
  2. Replace local-disk raw storage with MinIO or S3 for portability.
  3. Introduce a dedicated analytical layer such as DuckDB or ClickHouse for larger chart/query workloads.
  4. Add authentication, dataset ownership, and saved dashboards.
  5. Add load testing and formal observability.
